### 2026.03.09
* 基于“1小时局内关卡制肉鸽”方向，评估现有框架并补充架构建议（用于后续迭代对齐）

#### 现状结论（够不够用）
当前框架**适合继续做原型验证**，但如果要支撑“关卡池随机 + 资源经济 + 商店房 + 多敌人行为 + 最终Boss”的完整循环，建议从“单局战斗框架”升级为“局内流程框架 + 数据配置框架”。

已具备的基础：
1. 战斗主循环（索敌 / 移动 / 攻击 / 死亡）
2. 初始伙伴选择与开局流程
3. 进化时刻（可配置选项 + 选择后全队加成）
4. 可扩展攻击行为注册机制（适合继续扩展伙伴特色）

#### 建议补充的功能块（按优先级）
1. **RunState（局内状态）模块**
   - 职责：记录当前层数、营养、已拥有伙伴、可招募池、进化次数、当前关卡ID、随机种子。
   - 作用：把“单场战斗数据”与“整局肉鸽进度”解耦，避免逻辑散落在多个静态系统中。

2. **LevelNode / LevelPool（关卡节点与关卡池）**
   - 数据化定义关卡类型：普通战斗、精英、商店、事件、Boss。
   - 规则：每次通关后从待选池按权重抽下一关，支持保底与去重规则。
   - 产出：敌人编组、地形模板、奖励表、特殊规则。

3. **NutritionEconomy（营养经济）**
   - 统一“获得/消费”接口，支持来源标签（击杀、通关、事件、商店返利）。
   - 提供结算日志，便于后续做数值平衡和复盘。

4. **ShopSystem（商店房）**
   - 输入：RunState + 商店刷新规则。
   - 输出：可购买项（伙伴、进化时刻、一次性强化、回复等）。
   - 要点：价格曲线和刷新机制配置化，避免硬编码。

5. **EnemyBehavior扩展层（AI行为）**
   - 在当前攻击行为解耦基础上，再增加“移动/索敌策略接口”（近战追击、远程拉扯、隐身、冲锋、召唤等）。
   - 这样新敌人不必改动主循环，只需拼装策略。

6. **EncounterDirector（战斗导演）**
   - 关卡内负责刷怪波次、节奏控制、胜负判定条件（不只“全灭”）。
   - 为Boss关提供分阶段机制（血量阈值切换技能）。

7. **MetaConfig + Save（配置与存档）**
   - JSON/ScriptableObject统一管理伙伴、敌人、关卡、商店、奖励、数值曲线。
   - 增加最小存档：局外解锁、历史最佳、图鉴统计。

#### 建议的小型架构改动
1. 将 `LevelSystem` 的“经验升级”语义拆分命名（例如 CombatXP 与 RunProgress），避免和“关卡层数”概念冲突。
2. 降低全局静态系统耦合：保留静态入口，但把核心状态收敛到 `RunState` 实例，便于测试与回放。
3. 在 `UnitRuntimeData` 中逐步减少硬编码字段，补充“能力标签/行为配置ID”，为后续多样敌人留扩展位。
4. 在战斗结束处增加“奖励结算阶段”，不要直接进入下一场，便于插入商店/事件/进化奖励。

#### 推荐迭代顺序（两周可落地版本）
- 第1阶段：RunState + 营养经济 + 战斗后奖励结算UI。
- 第2阶段：关卡池随机（普通/商店/Boss）+ 商店房。
- 第3阶段：远程敌人与隐身敌人各1种 + Boss关原型。

以上改动可在不推翻现有战斗内核的前提下完成，属于“外层流程增强 + 配置化增强”。

#### 2026.03.09 实装补充（关卡配置化）
* 新增基于 `Resources/Configs/Levels/*.json` 的关卡出怪配置读取，每个 JSON 对应一个完整关卡。
* 关卡出怪字段包含：`time`（关卡开始时间）、`position`、`positionRandomRange`、`enemyTypeId`、`count`、`interval`。
* 新增 `EncounterDirector`，负责按时间推进刷怪：
  1. 到达 `time` 时开始该条目；
  2. 以 `interval` 为节奏生成，直到生成满 `count`；
  3. 生成坐标使用 `position + random(-range, range)`。
* `UnitManager` 接入战斗计时并在每帧驱动刷怪调度；开局改为优先加载配置关卡（默认 `关卡001`），并保留旧逻辑兜底生成蚊子。
* `EnemiesFactor` 增加 `CreateByTypeId`，支持通过敌人类型 Id 统一创建敌人实例。
